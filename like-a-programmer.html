// 01 全景

这堆词,各归各位

API、Token、agent、skill、vibe coding、养马养虾……它们不在一个层面上,而是一条链:从"原料"一路到"落地场景"。点任意一个词看它是什么。

1原料层资源与接口
LLM大脑本身(DeepSeek、Claude、GPT),被调用的那个模型。 API程序访问大脑的插座。人用聊天框,程序用 API。 Token燃料兼计价单位,也是"一次能记多少"的容量上限。 mac mini7×24 挂着跑任务的常驻机器;一台便宜安静的小服务器。 CLI命令行——能把动作写成脚本、批量重复的方向盘。
2组装层把能力组织起来
agent能自己规划、调工具、多步执行的"AI 员工",不是问一句答一句。 skill给 agent 的技能包/SOP;你自己封装的那套流程就是。 工作流把多个步骤和工具串成的自动化流水线。
3方法层人机分工的姿势
vibe coding你当产品经理、AI 当工程师,凭感觉快速迭代。你的主力方式。 vibe marketing同一套逻辑搬到营销:批量生成素材、快速试错。
4运行层你在"养"的 agent
养马 / HermesNous Research 的自托管自改进 agent,标语"与你一同成长"。有记忆、会自己长 skill,所以叫"养"。 养虾 / OpenClaw同赛道的另一个主流自托管 agent 框架(Claw→爪→龙虾)。
5应用场景由你的痛点决定,不是固定清单
做生意把能力包装成别人愿付费的产品。 写论文工具是研究的脚手架,不是产品。 做产品把痛点做成可复用的工具。 本职提效用 AI 解决你本行的真实痛点——别人替代不了的那部分。
// 02 真相一

程序员的一天:写代码只占一小块

这是最反直觉、也最该先知道的事。真正动手写代码,只占程序员一天的两到四成。剩下的才是"职业本体"。

一个程序员的典型一天 ↓
写代码
20–40%
读代码 · 理解老系统
调试
开会 · 对齐 · 做决定

读代码,而不是写代码。改一个老系统,八成功夫是搞懂它原本怎么跑——这块逻辑为什么这么写、那个三年前留下的诡异数据流为什么还在影响核心功能。

调试、对齐、做决定。排查只在生产环境复现的 bug、在会议里吵清楚"到底要做什么"、做架构取舍。AI 在系统设计、架构决策、理解业务上下文这些事上,依然很吃力。

所以 AI 啃掉的,是"写"这一小块里最机械的部分(样板、已知 bug、测试、文档)。留给人的,正好是不能自动化的那部分:判断、上下文、为结果负责。

// 03 真相二

AI 是不是泡沫:两件事同时为真

不是"是"或"否"。底层技术是真的,金融结构是虚的——这两件事同时成立。这个判断决定你愿意投多少进去。

真实的 / REAL

  • 技术有变革性,企业采用是真的
  • 生产力提升可测量
  • 领头公司有真实客户、真实收入
  • 基建会支撑未来几十年的应用

虚的 / FROTH

  • 估值远超任何近期盈利前景
  • 循环融资:英伟达↔OpenAI↔微软的钱在圈里转
  • 一批人靠"叙事"而非产品吃饭
  • 修正一旦来,会很猛
历史参照:亚马逊从 1999 年高点跌了 93%,然后成长为世界最值钱的公司之一——押对技术是必要的,但不充分,估值和时机同样要命。对你这种用 AI 干实事的人:泡沫在估值和叙事里,而你工具层亲手拿到的效用(Cursor 帮你把东西真做出来)是真金白银、已兑现。赌能跟着你走的技能,别赌周期。
// 04 核心

程序员到底怎么用 AI

关键不是"会写咒语"。他们用得好,是因为有判断力去指挥和验收它。这七条,就是你要长出来的肌肉。

当它是手快的初级工程师,不是神谕。任务切小,逐行验收;它会犯错,你得盯。
喂对上下文。给相关文件 + 清楚规格 + 规则文件,而不是把整个 repo 砸过去。
保持紧的反馈环。小 diff → 读 diff → 跑测试 → 不对就打断重来。
让它读代码,不只写代码。"解释这段""追踪这条数据流"——AI 是阅读加速器。
苦力丢给它,架构判断留给自己。样板、迁移、测试交出去;要建什么、怎么权衡你来定。
建护栏。测试、类型、lint——让 AI 的错误被自动接住,而不是靠你肉眼。
底子要够。你得看得出它什么时候在胡说(比如建议把密钥硬编码进代码)。

// 06 必修第一项 · 详解

读代码的能力,到底是什么

先打破最大的误解:读代码不是从第一行看到最后一行、每个字符都看懂——那是会把人读到崩溃的新手读法。真正的读代码是带着一个问题,去追一条线索。它更像审讯,不像阅读。

这套逻辑不止属于程序——几乎所有行业的专家都在干同一件事。侦探顺着一条证据往下追,技师靠一个异响定位故障,审计员盯着一个数字溯源……他们都不从头读到尾,而是带着问题进去、跳过噪音、在异常处停下、追问"为什么"。对象在变,动作不变。读代码,只是把这套你在别处早就有的本能,换个对象用。

拆开看,它是四个分动作

先看地图,别急着进门。读文件先看它叫什么、管什么;读函数先看名字和参数——在读函数体之前就先立一个假设。好代码的名字本身就是说明书。
追控制流:会跑哪条路。代码有分叉(if、try/catch、循环)。你追的是"如果发生 X,会走哪一条路、按什么顺序"。
追数据流:一个数据的旅程。挑一个变量,跟着它走:在哪儿被造出来、传给谁、在哪儿被改、最后去了哪。
认模式:把一坨当成一个。看多了,你会把"遍历列表""调接口再处理返回""提前返回的守卫"当成一个整体来读——像看波形,而不是看一个个小格子。

一段真实代码,看"读"的动作

一个常见的"调 LLM + 缓存"函数
async function generateSummary(docId) {
  const cached = await cache.get(docId);
  if (cached) return cached;

  try {
    const result = await callLLM(docId);
    await cache.set(docId, result, { ttl: 86400 });
    return result;
  } catch (err) {
    return fallbackSummary(docId);
  }
}
第一步 · 只看签名
给个 docId,异步,大概返回一段摘要。假设先立住,人还没进门。
第二步 · 追控制流
有缓存就直接返回(守卫式提前返回);否则进 try——说明里面有件会出错的事(callLLM,网络调用)。顺利:调 LLM → 存缓存 → 返回;失败:退回 fallback。读到 86400 停半秒换算 = 24 小时。
第三步 · 一句话复述
"有缓存用缓存;没有就问 LLM 并存一天;LLM 挂了就退回简单备用摘要。"能说出这句,这段就被你吃掉了。
第四步 · 问意图与后果
为什么缓存?LLM 又慢又贵。删掉 try/catch 会怎样?接口一抖,整个功能给用户崩了。再看它缺了什么:没校验 docId——而"漏掉边界检查"正是 AI 生成代码最常见的毛病。

读到第四步,你已经不只是看懂,而是抓到了 AI 的疏漏。这就是为什么读代码是必修第一项——它本质上就是"监督 AI"这个动作本身。

怎么练(这部分最关键)

预测—验证循环。 ★ 最该天天做先自己说出"我觉得它在干嘛、给这个输入会发生什么",再让 AI 解释。差在哪儿,你的盲区就在哪儿。用 AI 训练你,而不是替代你。
冷读自己的旧代码。打开一个 vibe 出来但没细看的文件,不运行、不问 AI,给每个函数写一句"它干嘛的",再让 AI 核对。
追一条线索。定个具体问题:"点提交之后到底发生了什么?"找到入口函数,顺着调用链一路追到底,在纸上画出来。
"删了会怎样"。指着某几行,先预测删掉的后果,再去验证。专练第四层的意图理解。
// 一条贯穿所有练法的总规矩

永远带着一个问题去读,绝不从头读到尾、指望它自己渗进脑子。没有问题,眼睛会滑过去、什么都留不下;有了问题,代码会自己浮现出你要的那条线。

// 07 实战

GitHub:那面墙的索引牌,你只用三个旋钮

那个吓人的筛选面板不是待办清单,只是图书馆的检索系统。搜 Code 是开消防栓(几十万条),搜 Repositories 是给你一摞书。永远从 Repositories 开始。

范围
Repositories
找项目来读、来学、来用。九成时候用这个。
语言
你的栈
HTML / JS / TS / Python / Vue,其他是噪音。
质量
stars + pushed
星标=口碑代理;最近推送=这项目还活着吗。

老手很少点筛选框——他们直接在搜索框打指令,更快。抄下来就能用:

GitHub 搜索语法
# 找还活着的、有口碑的 Vue 项目
> language:Vue stars:>500 pushed:>2026-01-01

# 直接搜你那套框架里的好项目,照着学结构

# 按你所在领域的标签逛开源项目(换成你的领域)

一个被低估的金矿:Issues 和 Pull requests。用某个库踩了坑,去它的 Issues 搜一下,常发现别人早踩过、还给了解法;读 PR 能看到真实的工程对话——比读成品代码更能学到"工程师怎么思考"。

// 08 下一步

你的路线(顺序很重要)

不是"从 GitHub 入手"——你已经在里面了。下一步是从"生成代码"转向"读懂代码"。按这个顺序走。

本周
用 uni-app stars:>200 找一个比你自己做过的项目简单、但结构干净的项目,clone 下来,让 AI 带你逐文件走一遍——看人家怎么组织目录、拆组件、调云函数。
然后
把 Git 用对。给你自己的一个项目建个分支做实验,练 commit、回滚——直到你"敢"乱改而不怕弄坏。
然后
挑你自己项目里一个"似懂非懂"的模块,关掉 AI,自己读懂、并能改动它。这一步是真正的过坎。
持续 · 碎片时间
补命令行和 HTTP / API 基础。不求精通,求遇到问题不抓瞎。
持续 · 给你的工具
养成验证习惯:边界情况、API 失败兜底。有真实用户的工具,出错代价更高。
// 终点检验

当你能在不靠 AI 的情况下,说清楚自己某个项目每一块在干嘛、改哪儿会影响哪儿——到这一步,你就已经"像程序员一样"在用 AI 了。

// 收尾 · 别忘了

工程是为目标服务的,不是来抢位置的

你把本职做到极致(Goal A)排在工程深度(Goal B)之前。所以工程只要补到两件事为止:能把自己的工具做可靠、能自给自足不被卡死。再往上为工程而工程,边际回报对你是递减的。

因为你真正稀缺的护城河,是你深耕的那个本行领域。一个真懂一个专业领域、又能动手建工具的人,比"又一个优秀工程师"罕见得多。

那些大神,每个人的第一个 commit 都和你的一样小、一样笨拙。门槛不在起点——起点是平的。门槛在你会不会一直走下去。而这件事只取决于你。
