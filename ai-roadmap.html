Rx ① 治本 · 工程纵深

主疗程 · 70%

把 GitHub 从“天书”变成“工作台”。不要试图读懂每一行——像看病历一样先抓主干。

1 从「用仓库」到「读仓库」
用任何开源项目前，先读它的 README
只问一件事：它解决什么问题。看主诉，不看每一行。
看项目的文件夹结构
文件怎么组织 = 架构思维。这是你做 CaseLog/PassRound 最该偷师的部分。
翻最近几个 commit 和 issue
真实工程怎么演进、怎么吵架、怎么取舍——一手的、活的工程现场。
2 从「读」到「改并提」
挑一个你在用、规模不大的项目
门槛越低越好。目标是走通流程，不是写惊艳代码。
做一次最低门槛贡献
改文档错别字、补中文说明、修一个小 bug——任选其一。
提出你的第一个 PR，走完整闭环
fork → 分支 → commit 规范 → PR → review → （被拒）→ 改 → 合并。走通一次，GitHub 就变工作台了。
3 补「手动挡」CS 底层（按你的开发卡点排序）
Git / 版本控制原理
立刻就用。分支模型、merge vs rebase、冲突解决——不是背命令，是理解它在干嘛。
一门语言吃透（建议就 JS 或 Python）
别贪多。选你天天写的那门，从语法走到“它为什么这样设计”。
HTTP 与 API 怎么工作
你 CaseLog 的云函数全靠这个：请求/响应、状态码、鉴权、超时。
数据库基础
表怎么设计、查询为什么慢。够支撑 CaseLog 数据与你的研究方向即可。
算法 / 操作系统 → 暂缓
你不是要刷题进大厂，往后放。先把上面四块变成肌肉记忆。

Rx ② 随访 · 紧跟 AI

维持 · 30%

三个动作就够，且必须低成本——不能让跟踪吃掉纵深时间。

核心原则：模型、工具、框架每月换一茬，今天的技巧明年就过时。Git 怎么工作、系统怎么分层，十年不变。底层是根，AI 跟踪是叶。
A 信息源做减法
停刷碎片化「AI 日报 / 又一个神器」
那是噪音。锁定 2–3 个高信噪比来源即可。
固定几个真正在做事的人的 GitHub 动态 + 官方发布
Anthropic / OpenAI 官方 + Hacker News 的 AI 板块，足矣。
B 用项目代替资讯
⇄ 这一步与 Rx① 重合
想懂某领域趋势，别读十篇测评——去 GitHub 找该领域 star 增长最快的项目，读 README 和近期 issue。这才是一手的工程趋势。
每月锁定一个高增长开源项目，读懂它
跟踪 AI 和深入 GitHub 是同一个动作——一举两得。
C 亲手做，不亲手追
新东西的唯一判断标准：能否解决你的具体问题
能解决 CaseLog / PassRound 某个痛点 → 花一下午试；不能 → 记个名字就过。

Rx ④ 手法 · 像程序员一样用 AI

操作技术

程序员用得好，不是 AI 技巧高，是判断力强——知道该信什么、改什么、拒什么。核心差异：把 AI 放进工程流，而不是当聊天框。

1 喂上下文，再提问
提问前先想：“它要看到什么才能答对？”
在 Cursor 里让 AI 读整个库，而不是复制片段出来问。主动喂相关文件、报错、项目约定——答案精度差一个量级。
2 AI 做脏活，人定架构
重复劳动甩给 AI，设计判断自己留着
写测试、改样板、重构、翻译报错 → 交给 AI。怎么分层、要不要这个依赖、设计对不对 → 自己定。AI 写“怎么做”，你定“做什么、为什么”。
3 永远 review，从不盲信
AI 生成的代码，逐行读懂再用
资深程序员用得快，是因为能一眼看出坑和幻觉。你判断力还在长——所以每次 review 都当成读源码训练，这恰恰是 Rx① 的纵深。
4 把 AI 当源码私教
看不懂的 GitHub 代码，贴给 AI 逐行讲
“逐行解释这在干嘛、为什么这么写。”望而生畏的天书 → 带私教的源码课。这一条直接打通你“深入 GitHub”的卡点。
5 git / 终端 / 报错，边用边问
不只问“怎么做”，追问“为什么是这样”
写 commit、解释报错堆栈、查 git 操作——随手问。把你正在补的 Git/HTTP 底层，变成边用边学。工具即老师。
一句话：你想“像程序员一样用得好”，真正要补的不是 AI 技巧，是 Rx① 那套纵深。你越懂底层，AI 在你手里就越强——这俩是同一件事。
